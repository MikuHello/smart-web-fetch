# 文档风格规范

本文档定义仓库内长期文档的写法边界。目标是让文档始终面向当前使用和当前维护，而不是持续携带历史包袱。

## 1. 适用范围

本规范适用于以下长期驻留文档：

- `README.md`
- `README_EN.md`
- `skills/*/SKILL.md`
- `spec/*.md`

本规范不适用于以下变更沟通载体：

- Pull Request 描述
- commit message
- review comment
- issue 讨论

这些载体可以记录变更背景、迁移原因和兼容性影响。

## 2. 核心原则

- 长期文档只描述当前事实、当前 contract、当前结构和当前约束。
- 长期文档应帮助读者完成今天的使用或维护任务。
- 对理解今天的使用和维护没有直接帮助的历史背景，默认不进入长期文档。
- 防回归信息优先落到测试，变更背景优先落到 PR / commit / review。

## 3. 文档分类要求

### 3.1 README

README 只应包含：

- 当前用途和定位
- 当前安装方式
- 当前调用方式
- 当前参数、输入、输出和示例
- 当前依赖和运行前提

README 不应包含：

- 迁移历史
- 旧实现回顾
- 兼容旧写法提醒
- reviewer 才需要的背景解释
- 仅为了防回归而保留的历史注释

### 3.2 SKILL.md

`SKILL.md` 只应包含：

- 触发场景
- 调用方式
- 必要参数和行为
- 运行前提

`SKILL.md` 应尽量短，避免携带额外背景信息。

`SKILL.md` 不应包含：

- 仓库历史
- 迁移说明
- 旧入口或旧参数的兼容提醒
- 与触发和使用无关的维护细节

### 3.3 spec 文档

`spec` 文档只应包含：

- 当前行为契约
- 当前结构约束
- 当前测试基线
- 当前实现同步要求

`spec` 文档不应写成：

- 变更日志
- 迁移手册
- 旧方案对照表
- 历史背景档案

若某项背景信息不会影响今天的维护决策，应删除。

## 4. 允许写什么

以下信息可以保留在长期文档中：

- 当前明确支持的参数、接口、入口
- 当前必须满足的前提条件
- 当前失败行为和错误约束
- 当前目录结构与职责边界
- 当前测试覆盖范围

保留标准只有一个：删除这段信息后，是否会影响今天的使用或维护决策。

## 5. 默认删除的内容类型

除非会直接影响当前使用或维护决策，否则以下内容默认删除：

- “不再”
- “旧的”
- “曾经”
- “历史遗留”
- “已移除”
- “兼容旧写法”
- “从 X 改为 Y”
- “避免回到旧方案”
- “为了解决之前的某问题”
- “此前/原先/过去”

判断标准不是措辞本身，而是该内容是否在解释过去。

## 6. 写作方式

- 优先使用正向描述，避免用“不是 X”来定义“现在是什么”。
- 优先写当前 contract，不写当前 contract 如何取代旧 contract。
- 能用一句当前事实表达清楚时，不追加历史背景句。
- 若某信息仅对 reviewer 有价值，应移入 PR 描述，而不是长期文档。

示例：

- 好：`运行时无需 curl / jq / perl。`
- 差：`不再依赖旧版实现中的 curl / jq / perl。`

- 好：`scripts/ 目录只保留平台入口，业务实现位于 main.py 与 core/。`
- 差：`scripts/ 目录不再承载旧的业务实现。`

## 7. 提交前复查

修改长期文档后，提交前必须做以下检查：

1. 这段内容是在描述当前事实，还是在解释过去？
2. 删除这段内容，是否会影响今天的使用或维护决策？
3. 这段内容是否更适合放进 PR / commit / review？
4. 这段内容是否只是为了提醒不要回到旧方案？

若第 2 项答案为“不会影响”，默认删除。

## 8. 与测试和变更沟通的分工

- README / `SKILL.md` / `spec`：描述当前 contract
- tests：防止行为回退
- PR / commit / review：解释为什么改、从哪里改来、兼容性如何变化

不要让长期文档承担测试和变更说明的职责。
